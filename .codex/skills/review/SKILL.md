# Review

프로젝트의 변경 사항을 아키텍처·기술 스택·CRITICAL 규칙 관점에서 점검하고 표 형식으로 결과를 보고한다.

## When to Use

- PR을 머지하기 전 변경 셋의 규약 준수 여부를 확인할 때.
- 큰 리팩터/기능 추가 직후 의도한 범위를 벗어나지 않았는지 점검할 때.
- 새로 합류한 사람이 직전 변경이 ADR/ARCHITECTURE/CRITICAL 규칙을 따르고 있는지 빠르게 검증해야 할 때.

## Inputs

- (선택) 검사 대상 변경 범위. 미지정 시 작업 트리의 모든 변경(uncommitted + 현재 브랜치 vs `main`)을 검사한다.
- 다음 문서가 레포에 존재해야 한다:
  - `/CLAUDE.md` 또는 `/AGENTS.md`
  - `/docs/ARCHITECTURE.md`
  - `/docs/ADR.md`

## Workflow

1. 위 세 문서를 먼저 읽어 아키텍처 의도, 기술 스택 범위, CRITICAL 규칙을 파악한다.
2. 변경된 파일 목록을 확인한다 (입력으로 범위가 주어지면 그 범위, 아니면 `git status` + `git diff main...HEAD`).
3. 변경 셋을 다음 체크리스트로 검증한다:
   1. **아키텍처 준수** — ARCHITECTURE.md에 정의된 디렉토리 구조와 모듈 경계를 따르는가?
   2. **기술 스택 준수** — ADR에 정의된 기술 선택을 벗어나지 않았는가? 새 의존성이 추가됐다면 ADR에 부합하는가?
   3. **테스트 존재** — 새 기능/수정에 대응하는 테스트가 작성/갱신되어 있는가?
   4. **CRITICAL 규칙** — `CLAUDE.md`/`AGENTS.md`의 CRITICAL 항목 중 위반된 것이 있는가? (예: privacy 판정의 단일 책임, allowlist 외 frontmatter 노출, `private/**` tripwire, transclusion 규칙 등)
   5. **빌드 가능** — 프로젝트 빌드 명령이 에러 없이 통과하는가? 가능한 경우 실행하여 확인한다.
4. 아래 출력 형식대로 표를 작성한다.
5. 위반 항목이 있으면 각각에 대해 **무엇을 어떻게 고쳐야 하는지** 구체적인 수정안을 제시한다.

## Outputs

| 항목 | 결과 | 비고 |
|------|------|------|
| 아키텍처 준수 | ✅/❌ | {상세} |
| 기술 스택 준수 | ✅/❌ | {상세} |
| 테스트 존재 | ✅/❌ | {상세} |
| CRITICAL 규칙 | ✅/❌ | {상세} |
| 빌드 가능 | ✅/❌ | {상세} |

위반이 있으면 표 아래에 항목별 수정 방안을 제시한다.

## Next Steps

- 위반 사항이 있으면 수정 후 review 스킬을 다시 호출해 재검증한다.
- 모든 항목이 ✅이면 머지 또는 다음 phase로 진행한다.
