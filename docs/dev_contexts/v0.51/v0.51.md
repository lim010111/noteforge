# v0.51 — Dev image picker: 파일 업로드 지원

## 구현 결과

- 브랜치: `feature/v0.51-dev-image-upload`
- 결정: URL 입력 대신 **파일 저장 + frontmatter 갱신 + pipeline cache refresh**를 하나의 dev-only 업로드 트랜잭션으로 구현한다.
- 새 endpoint: `POST /__obpub/upload-attachment` (`multipart/form-data`)
- 새 설정: `attachments.uploadDir` 기본 `"attachments"`, `attachments.uploadMaxBytes` 기본 `10_485_760`
- closure 변경: 공개 노트의 `cover`/`thumbnail` frontmatter가 `/attachments/<rel>`이면 본문 임베드 없이도 public attachment closure에 포함된다. private 노트의 frontmatter 참조는 `publicNoteIds` gate 때문에 closure에 기여하지 않는다.
- UI: 각 `cover`/`thumbnail` 슬롯에 현재 선택 미리보기, embedded image 후보 grid, drop zone, 적용/초기화 버튼을 제공한다. 후보 grid 클릭은 미리보기만 바꾸고, `적용` 버튼이 frontmatter 저장을 수행한다. URL 직접 입력은 제거했다.
- Migration note: 기본 업로드 위치가 vault-relative `attachments/`이므로 업로드 결과 URL은 `/attachments/attachments/<file>` 형태가 될 수 있다. 다른 vault 폴더를 원하면 `attachments.uploadDir`를 POSIX 상대 경로로 지정한다.

## 원 계획
                                                                                                  
 Context                             

 문제. 현재 dev-image-picker__trigger("이미지 설정") 버튼은 두 가지 입력만 받음:
 1. 본문에 이미 임베드된 이미지(embeddedImages) 그리드에서 선택
 2. URL 입력칸에 https://… 또는 /attachments/<rel> 직접 붙여넣기

 → 즉 새 이미지를 cover/thumbnail로 쓰려면 사용자가 (a) Obsidian 안에서 vault 어딘가에 파일 복사 → (b) 본문에 ![[…]] 
 임베드 → (c) 서버 재기동/다시 로드 → (d) 그제서야 picker 그리드에서 선택 의 4단계를 밟아야 함. 도그푸드를 하면 이
 마찰이 모든 글마다 반복됨.

 원인 (UX 외 구조적 제약 1개 동반): packages/core/src/pipeline.ts:279-296 의 collectAttachmentRefs 는 노트 본문만
 스캔. 따라서 frontmatter cover: /attachments/foo.png 만 단독으로 있을 경우 closure에 안 들어가고,
 sanitizePublicImageFrontmatter (pipeline.ts:649) 가 strip 해버림. 이건 picker 의 URL 붙여넣기 모드도 사실상 "본문에
 한 번이라도 인용된 이미지"만 동작한다는 뜻이며, 단순 업로드만 추가해서는 작동하지 않음.

 의도한 결과.
 - (1) Picker 다이얼로그 안에서 로컬 파일을 드래그-드롭 / 파일 선택 / 클립보드 붙여넣기 로 바로 업로드 → vault 의 지정
  폴더로 사본 저장 → cover/thumbnail frontmatter 자동 set 까지 하나의 트랜잭션으로 처리. 본문 임베드 단계 불필요.
 - (2) frontmatter cover/thumbnail 단독 설정도 정상 노출되도록 closure 빌더 보정 (privacy 계약은 그대로 유지: private
 노트의 frontmatter ref 는 closure 에 기여하지 않음).
 - (3) 기존 임베드 그리드 / URL 붙여넣기 / 초기화 버튼은 그대로 유지 (escape hatch).

 Branch

 결정 (모두 사용자 확인 ✓)

 D1. 업로드 후 cover 까지 한 번에 처리하는 단일 엔드포인트

 새 엔드포인트 POST /__obpub/upload-attachment (multipart/form-data) 가 파일 저장 + frontmatter 갱신 둘 다 원자적으로
 수행. 이유:
 - 두 endpoint 분리 시 race + 부분 실패 (파일은 vault 에 들어왔는데 frontmatter 는 갱신 실패) 가능
 - 기존 POST /__obpub/cover 는 그대로 둠 (그리드 선택 / URL 붙여넣기 / clear 모두 계속 사용)
 - 실패 시 업로드된 파일을 rollback (delete)

 D2. 업로드 위치

 vault-relative attachments/ (root level), 충돌 시 suffix -1, -2, …. 즉 Obsidian 의 일반적 attachment 폴더 컨벤션.
 - 새 config 필드 attachments.uploadDir (string, default "attachments") 추가 → 사용자가 원하는 위치로 변경 가능 (검증
 규칙은 D9)
 - slug 기반 서브폴더는 하지 않음 (Obsidian 기본 동작과 다르고 vault 검색 시 산만함)
 - 파일명 sanitization 정책은 D8 참조

 D3. closure 빌더 확장 (privacy-adjacent)

 pipeline.ts 에서 공개 노트의 allowlisted cover/thumbnail frontmatter 값 도 attachment ref 로 수집. 사설 노트는 기존과
  동일하게 본문 ref만 수집(또는 frontmatter ref 도 수집하되 closure builder 가 어차피 public source 만 카운트). 핵심:
 - 새 ref 의 sourceNoteId = 노트 slug
 - attachment ref 발견 위치: frontmatter.cover / frontmatter.thumbnail 가 /attachments/<rel> 형태일 때만. http(s) 외부
  URL은 무시.
 - private 노트 frontmatter 는 어차피 attachmentFilter 가 public source 로 필터하므로 closure 기여 없음 — 하지만
 canary 로 명시 검증.
 - 이 변경은 packages/core 안이지만 src/privacy/ 디렉토리는 안 건드림. privacy 결정 지점은 그대로 (classify /
 attachmentFilter / sanitize). 단지 ref 수집을 한 군데 추가.

 D4. 보안 / 검증 (서버)

 - Dev only: middleware 등록 자체가 astro:server:setup 안에서만 일어나므로 production build 에는 노출되지 않음.
 - Same-origin: 기존 isSameOrigin() 재사용.
 - multipart 파서: D7 (수작업, 외부 dep 없음).
 - MIME 검증: image/png|jpeg|gif|webp|svg+xml (확장자 → MIME 매핑 + 파일 매직 바이트 sniff). PDF 는 cover 용으로 안
 받음. (단 attachments.allowedExtensions 에 .pdf 가 있어도 cover 용 업로드는 image only.)
 - 사이즈 제한: 10 MB (config attachments.uploadMaxBytes, default 10_485_760).
 - 경로 traversal: path.resolve(vaultRoot, uploadDir, sanitizedFilename) → isInside(vaultRoot, …) guard.
 - slug / field: 기존 isSafeSlug + field 는 'cover' | 'thumbnail' 만.
 - rollback: frontmatter write 가 실패하면 방금 쓴 파일 unlink. 단 동명 기존 파일을 덮어쓴 경우는 rename 으로 backup →
  실패 시 복원.

 D6. Race condition: 응답 전에 pipeline cache 동기 refresh (CRITICAL)

 문제. integration.ts:222-249 의 onInvalidate 는 void refreshPipelineCache() (fire-and-forget) 후 즉시
 server.ws.send({type:'full-reload'}) 를 발사. 즉 watcher 가 frontmatter 수정으로 invalidate 되면, refresh 가 끝나기
 전에 브라우저가 reload. 기존 dev cover 는 attachment 가 이미 closure 에 있어서 무관했지만, 신규 업로드는 closure 에
 한 번도 들어간 적 없는 파일이라 reload 직후 /attachments/<new-file> 가 404.

 해결. devUploadMiddleware 가 다음 순서로 동작:
 1. 파일 저장
 2. frontmatter write
 3. await refreshPipelineCache() — 새 helper 를 DevUploadMiddlewareDeps 에 주입. integration.ts 에서 closure 위에
 정의된 동일 함수를 직접 export 해서 전달.
 4. JSON 200 응답 {ok:true, url:'/attachments/<rel>'}
 5. 클라이언트는 응답 받고 location.reload() 호출 (HMR 신호와 무관하게 능동 reload)
 6. watcher 의 후속 HMR full-reload 는 어차피 idempotent — 이중 reload 가 한 번 발생할 수 있지만 정상 동작 (dev only).

 registerDevUploadMiddleware 시그니처에 refreshPipelineCache: () => Promise<void> 추가. 기존 getPipelineSnapshot 도
 같이.

 D7. multipart/form-data 파서 — 수작업, dep 추가 없음

 제약. Vite dev middleware 의 req 객체는 Connect-style AsyncIterable<Uint8Array | string> (현재
 devCoverMiddleware.ts:21 참조). Request.formData() API 는 가용하지 않음.

 선택. 외부 의존성 추가 대신 최소 수작업 파서:
 - Content-Type: multipart/form-data; boundary=... 헤더에서 boundary 추출
 - 전체 body 스트림 → Buffer 누적, MAX 사이즈 (uploadMaxBytes) 초과 즉시 reject
 - Buffer 에서 --<boundary> 로 split → 각 part 의 헤더(Content-Disposition, Content-Type) 파싱 + content
 - 필드 파일은 1개만 허용 (field=file), 나머지 텍스트 필드(slug, field)
 - 80~120 줄 분량, parseMultipart() helper 로 분리 + 단위 테스트로 boundary 변형 / 잘못된 헤더 / 사이즈 초과 / 빈 part
  / 다중 파일 등 케이스 커버

 이유: privacy-first 제품에 dev-only 기능을 위해 sup-chain 위험 감수할 필요 없음. 코드 분량은 검토 가능한 크기.

 D8. 파일명 sanitization — Unicode 허용, 제어/경계 문자 차단

 - 허용: 한글 등 Unicode (예: 스크린샷.png)
 - 차단: NUL (\0), 경로 구분자 (/, \), .., 제어 문자 (0x00-0x1F, 0x7F), CSS/HTML 인젝션 가능 문자 (', ", <, >, 백틱)
 - 정책: 차단 문자는 _ 로 치환, 다중 공백/언더스코어는 1개로 압축, 양 끝 공백/점 trim, 빈 문자열/./.. 결과는
 image-<timestamp>.<ext> 로 fallback
 - 확장자: 클라이언트 제공 확장자 + 매직 바이트 sniff 둘 다 일치 + attachments.allowedExtensions 안에 있어야 통과.
 불일치 시 매직 바이트 우선.
 - 충돌: <base>.<ext>, 존재하면 <base>-1.<ext>, <base>-2.<ext>, … 최대 999, 그 위는 timestamp 접미사
 - CSS 주입 방어: Note.astro:42 의 background-image: url('${note.heroImage}') 에서 값에 ' 가 들어가면 break 가능 →
 sanitization 으로 차단되므로 안전. 추가로 component 측 escape 강화는 별도 PR 가능.

 D9. uploadDir 검증 (zod refine)

 attachments.uploadDir:
 - 빈 문자열 차단
 - 절대경로 (/..., C:\...) 차단
 - .. 포함 차단
 - private 또는 private/ 로 시작하면 거부 (tripwire 보호) — 단 unsafeAllowPrivateFolder: true 면 허용
 - POSIX-style 만 허용 (\ 차단)
 - 디폴트 "attachments". 디렉토리가 없으면 mkdir -p

 D10. unsafeAllowPrivateFolder / private 슬러그 보호

 - 업로드 대상 디렉토리가 private/ 로 떨어지지 않도록 D9 에서 차단
 - request 의 slug 가 publicSlugs 에 없으면 (이미 구현된 unknown_slug 흐름) 거부 — 즉 private 노트 cover 업로드 불가
 - frontmatter 단독 cover 가 closure 에 들어가게 되는 D3 의 변경은 attachmentFilter 의 publicNoteIds gate 로 private
 노트는 자동 제외 → canary 검증 필수

 D11. 동시성 / 멱등성

 - 같은 slug 에 대한 동시 업로드 — 단일 사용자 dev 환경 가정, last-write-wins. 락 미사용.
 - dedupe (해시 비교) 미사용 — 항상 새 파일 생성 (suffix), 예측 가능.
 - 0-byte 파일 업로드 reject.
 - 업로드된 파일 SVG 의 임베드 스크립트 — 이미 시스템 전체가 SVG attachment 를 허용 중 (config.ts:29). 본 변경은 정책
 변동 없음. 사용자가 SVG cover 사용 시 동일한 위험. 별도 hardening 은 out of scope.

 위험 / 트레이드오프

 - closure 빌더 변경 (D3) 은 privacy-adjacent. 본문 ref 가 없는 cover/thumbnail attachment 가 새로 closure 에 들어가게
  됨 → 정확히 의도한 동작이지만, 기존 사이트가 빌드 결과의 dist/attachments/ 에 추가 파일을 보게 될 수 있음. 이는 leak
  이 아니라 사용자가 명시적으로 cover 로 지정한 attachment 만 추가되는 것이므로 안전. canary 테스트로 회귀 방지.
 - multipart 파서 직접 구현 (D7): 외부 dep 미사용 결정. boundary / size 한도 / 헤더 파싱 모두 단위 테스트로 커버. Vite
  middleware 가 향후 Web Request API 를 노출하면 그쪽으로 마이그레이션 가능.
 - 이중 reload (D6 부산물): 업로드 응답 후 location.reload() 와 watcher 의 HMR full-reload 가 거의 동시에 두 번 발생할
  수 있음. 시각적으로 깜빡일 수 있으나 정상 동작. 필요 시 watcher 의 invalidate 를 짧게 suppress 하는 후속 최적화.
 - SVG 임베드 스크립트: 시스템 전체가 SVG attachment 허용 — 본 변경은 정책 무변동. 사용자가 SVG cover 사용 시 동일
 위험 (브라우저가 <img src="…svg"> 로 로드하면 스크립트 실행 안 됨, 단 직접 navigation 시 실행). 별도 hardening 후속.
 - rollback 의 race: frontmatter write 후 unlink 전에 다른 요청이 같은 파일 참조를 추가하면 leftover ref 가능. 단일
 사용자 dev 환경 가정으로 무시. rollback 은 unlink 만, frontmatter 미터치.
 - production 노출 zero: 모든 새 코드는 astro:server:setup 안에서만 등록됨. 빌드 산출물에 endpoint 없음. config 필드만
  production schema 에 존재하지만 사용처는 dev 전용.

2026-05-02 follow-up: 실제 vault 적용 실패 수정

증상

- `test_imgs/` 의 이미지를 업로드해도 설정창이 닫히고 hero/thumbnail 변경이 보이지 않음.
- 기존 URL/후보 이미지 `적용`도 frontmatter 저장 이후 페이지에 즉시 반영되지 않음.

원인

- 업로드 성공 UI가 `dialog.close()` 를 먼저 호출한 뒤 reload 하도록 되어 있어, 사용자가 보기에는 파일 선택 직후 설정창만
  닫히는 흐름이었다.
- `/__obpub/cover` 는 frontmatter write 후 integration pipeline cache / Astro Content Layer refresh 를 기다리지 않았다.
- `/__obpub/upload-attachment` 도 integration 내부 cache 만 갱신했고, Astro Content Layer store 는 직접 갱신하지 않았다.
  vault 가 Astro `src/content` 밖에 있으므로 Content Layer refresh 를 명시적으로 호출하지 않으면 reload 후에도 이전
  `heroImage` / `thumbnailImage` 데이터를 볼 수 있다.

수정

- `astro:server:setup` 의 `refreshContent({ loaders: ['@noteforge/astro/loader'] })` 를 dev refresh 경로에 연결했다.
- watcher invalidation 은 pipeline cache + Content Layer refresh 완료 후 full-reload 를 보낸다.
- `/__obpub/cover` 도 refresh 실패 시 frontmatter 를 rollback 하고 `pipeline_refresh_failed` 를 반환한다.
- `DevImagePicker` 는 성공 시 설정창을 먼저 닫지 않고 `저장됨 · 새로고침 중` 상태 표시 후 `location.reload()` 만 호출한다.
- hidden file input 의 programmatic click 이 drop zone click handler 를 다시 타지 않도록 방어했다.

2026-05-02 follow-up: URL 입력 제거 + 후보 grid 미리보기화
