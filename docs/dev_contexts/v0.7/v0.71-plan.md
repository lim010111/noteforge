# v0.71 — Sidebar Categories 정렬·문서 카운트

## Context

`apps/blog`의 사이드바 Categories 섹션은 카테고리 모드(`hideLeafNotes: true`)에서 두 가지 UI 문제를 갖고 있다.

1. **정렬 깨짐** — 같은 depth의 형제 카테고리라도 하위 카테고리 유무에 따라 마크업이 갈린다 (`packages/theme-default/src/components/FolderTree.astro:55-64, 123-135`). 하위 카테고리가 있는 노드는 `<details><summary>▸ 이름</summary>...`로 chevron(1.25rem + gap)을 차지하지만, "leaf-category"(자식 없음, 노트만 보유 또는 비어있음)는 chevron 없이 `<a class="folder-tree__name">`만 렌더된다. 결과적으로 형제끼리 좌측 정렬이 어긋난다.
2. **카테고리당 문서 수 미노출** — 사용자는 카테고리 이름만 보고는 그 안에 문서가 몇 개인지 알 수 없다. `apps/blog/src/lib/folderAggregation.ts:211-215`의 `countNotesRecursive`는 이미 폴더-인덱스 페이지(`buildFolderIndexViewModel:280`, `buildCategoryIndexViewModel:355`)에서 자식 카드의 `noteCount`를 만들 때 쓰이지만, 사이드바 트리 노드(`FolderNode`)에는 카운트 필드 자체가 없다.

목표: 같은 depth 형제는 chevron 유무와 무관하게 시각적으로 정확히 정렬되고, 각 카테고리 옆에 자기 서브트리(자기 자신 포함, 자손 전부) 안에 있는 publishable 문서 수가 표시된다.

## 사용자 예시 재확인

```
PEFT (frontmatter category=PEFT/LoRA인 "LoRA 란" 1개만 존재)
  └ LoRA  →  1
PEFT 자체  →  1   (서브트리 안의 노트 1개)
news (category=news, 노트 3개)
  →  3
```

→ 카운트는 **재귀(서브트리 합계)**. 단순 `node.notes.length`가 아니라 `node.notes.length + Σ child.noteCount`.

## 접근

1. `FolderNode`에 `noteCount: number` 필드 추가(필수). 트리 빌더가 마지막에 단일 재귀로 채운다.
2. `FolderTree.astro`는 leaf-category에도 항상 chevron 자리(투명 placeholder)를 예약하여 형제 정렬을 보장하고, `noteCount > 0`이면 이름 우측에 카운트 배지를 출력한다.
3. CSS는 `.folder-tree__summary`의 flex 행에 `flex: 1; min-width: 0;`을 가진 이름 + 우측 카운트를 추가하고, leaf-category 행도 동일한 `[chevron-spacer][name][count]` 레이아웃으로 통일한다.
4. 기존에 `countNotesRecursive`를 호출하던 두 view-model(`buildFolderIndexViewModel`, `buildCategoryIndexViewModel`)은 새로 노드에 들어간 `child.noteCount`를 그대로 사용해 중복 계산을 제거한다.

## 브랜치

- 현재 `main`에 v0.71 브랜치가 없음. 첫 단계로 `git checkout -b v0.71`.

## 변경 대상 파일

### 1. `packages/theme-default/src/lib/folderTree.types.ts`

`FolderNode`에 필드 추가:

```ts
export interface FolderNode {
  name: string;
  path: string;
  children: FolderNode[];
  notes: { /* unchanged */ }[];
  /** 자기 자신 + 모든 자손 폴더의 publishable 노트 합계. 빌더가 항상 채움. */
  noteCount: number;
}
```

### 2. `apps/blog/src/lib/folderAggregation.ts`

- 기존 `countNotesRecursive` (211-215)를 `annotateNoteCounts`로 교체:

```ts
function annotateNoteCounts(node: FolderNode): number {
  let total = node.notes.length;
  for (const child of node.children) total += annotateNoteCounts(child);
  node.noteCount = total;
  return total;
}
```

- `buildFolderTree` 끝(현 121: `sortTree(root); return root;`)과 `buildCategoryTree` 끝(현 207: 동)에 `annotateNoteCounts(root);` 호출 추가. 빈 root도 `noteCount = 0`으로 초기화되어야 하므로 `root` 리터럴 정의(`{ name: '', path: '', children: [], notes: [] }`)에도 `noteCount: 0`을 명시 — `annotateNoteCounts`가 덮어쓰지만 strict TS에선 초기값 필요.
- 트리 안에서 새 노드를 만드는 두 자리(현 96-101, 181-187)에도 `noteCount: 0` 초기값 추가.
- `buildFolderIndexViewModel:280`과 `buildCategoryIndexViewModel:355`의 `noteCount: countNotesRecursive(child)`를 `noteCount: child.noteCount`로 단순화. 이로써 같은 트리에 대한 이중 재귀가 사라진다.

## 영향 범위 / 회귀

- `FolderNode`는 `apps/blog`와 `packages/theme-default`에서만 사용 (`@noteforge/theme-default` 배럴이 type-only로 export). 외부 패키지(`@noteforge/core`, `@noteforge/astro`, `@noteforge/cli`)는 import 안 함 — `pnpm -r typecheck`로 확인.
- `apps/blog/src/lib/categoryOverviewPayload.ts:16` `collectDescendantNotes(node)`는 노트 본체를 모으는 별도 traversal이라 noteCount와 무관 — 영향 없음.
- `noteCount`는 빌드 시 트리 단일 사후 패스로 한 번 계산 (`annotateNoteCounts`); 기존 `countNotesRecursive` 호출 2곳 제거로 동일 트리에 대한 이중 재귀가 사라져 미세 성능 개선.
- 정렬 fix는 순수 CSS + leaf-category markup 추가만 변경. `role="tree"` 미사용, `client:` 디렉티브 없음 — 기존 (9)(10) 어서션 영향 없음.
- privacy: 입력은 이미 `filterPublishable`로 처리된 트리만 카운트되므로 누출 없음. canary 테스트 (8)·빌드 audit canary 0회 회귀 확인.
- `shouldRenderAsLeafLink` 계약 유지: leaf-category는 여전히 토글 불가(빈 `<details>` 안 만듦). placeholder는 *시각적* 정렬만 보강하고 cursor·hover·focus는 일반 링크와 동일하게 유지 (`.folder-tree__chevron--placeholder { visibility: hidden; pointer-events: none; }` 추가 필요). 커밋 `7ee5439`의 "false toggle 어포던스 제거" 의도와 충돌하지 않음.
- BaseLayout은 sidebar를 desktop grid + mobile drawer 두 번 렌더 (`Sidebar.types.ts` 주석). 동일 마크업 양쪽 노출 — 모바일 drawer에서도 카운트 배지가 보이고, leaf-category 정렬도 동일하게 동작. 추가 작업 불필요.

## 범위 외 (Non-goals)

- 폴더 모드(`hideLeafNotes: false`) `.folder-tree__note-link`의 margin 정렬 — 기존 동작 유지.
- chevron 토글 영역의 클릭 가능 영역 확대 / 키보드 행동 변경.
- 카운트 배지 클릭으로 카테고리 필터 등 인터랙션 — 정적 표시만.
- 카테고리별 색상 dot 부활 (v0.5에서 의도적으로 제거된 결정 유지).

수동 시각 회귀 시 vault에 다음과 같은 노트가 있는지 확인:
- frontmatter `category: PEFT/LoRA` 1개 → PEFT(1), LoRA(1)
- frontmatter `category: news` 또는 `category: news/...` 여러 개 → news(N)
- 카테고리 미설정 또는 빈 카테고리 → root.notes로 가서 사이드바 트리에는 안 나옴 (`hideLeafNotes`).
